<h3 id="1-자바-내장-유틸리티-stringtokenizer와-wrapper-클래스">1. 자바 내장 유틸리티: StringTokenizer와 Wrapper 클래스</h3>
<p>문자열을 특정 구분자로 쪼개거나, 기본형(Primitive type) 데이터를 객체(Object)처럼 다루기 위해 제공되는 필수 API다.</p>
<ul>
<li><strong>StringTokenizer:</strong> <code>String</code>의 <code>split()</code> 기능과 유사하게, 하나의 긴 문자열을 지정한 구분자(기본값: 공백)를 기준으로 여러 개의 토큰(Token)으로 분리할 때 사용한다.</li>
<li><strong>Wrapper 클래스:</strong> <code>int</code>, <code>double</code> 등의 기본형 값을 힙(Heap) 메모리에 할당되는 '객체'로 포장(Boxing)하여, 클래스에 정의된 유용한 메서드(ex. <code>Integer.parseInt()</code>)나 상수를 활용할 수 있게 해준다.</li>
</ul>

        Integer objNum = 10;

<h3 id="2-예외-처리의-기본-체계-try-catch-구문과-다중-catch-규칙">2. 예외 처리의 기본 체계: try-catch 구문과 다중 catch 규칙</h3>
<p>프로그램 실행 중 발생하는 비정상적인 상황(Exception)을 감지하고, 시스템이 강제 종료되지 않도록 안전한 우회 경로를 제공하는 문법이다.</p>
<ul>
<li><strong>try-catch-finally 구조:</strong> 예외 발생 가능성이 있는 코드를 <code>try</code> 블록에 넣고, 예외가 터지면 <code>catch</code> 블록이 이를 잡아 처리한다. <code>finally</code>는 예외 발생 여부와 상관없이 무조건 실행되는 블록으로, 주로 점유했던 메모리 자원(네트워크, DB 등)을 반납(Close)할 때 사용한다.</li>
<li><strong>다중 catch와 다형성 규칙:</strong> 하나의 <code>try</code> 블록에서 여러 종류의 예외를 처리할 수 있다. 이때 부모 클래스인 <code>Exception</code> 하나만 쓰면 모든 예외를 다 잡을 수 있지만, 실무에서는 <strong>어떤 예외인지 구체적으로 파악하기 위해 상세한 예외부터 처리하는 것을 권장</strong>한다.</li>
</ul>
<blockquote>
<p>🌟 <strong>다중 catch 블록의 순서 규칙 **
다중 catch 구문을 작성할 때는 반드시 예외 클래스의 계층 구조상 **하위(자식) 예외부터 먼저 지정</strong>해야 한다. 포괄적인 최상위 부모인 <code>Exception</code>을 먼저 작성하면 모든 예외가 그곳으로 빨려 들어가버려 뒤쪽의 상세 catch 문이 도달할 수 없는 데드 코드(Unreachable code)가 되기 때문이다. (<code>Exception</code>은 맨 마지막에 배치하는 것이 관례)</p>
</blockquote>
<pre><code class="language-java">public class ExceptionBasic {
    public void calculate() {
        try {
            int n = 0;
            int result = 10 / n; // 0으로 나누어 ArithmeticException 발생

            String s = null;
            System.out.println(s.length()); // NullPointerException 발생 지점

        } catch (ArithmeticException e) {
            // 하위 구체적인 예외 1
            System.out.println(&quot;수학적 오류 발생: &quot; + e.getMessage());
        } catch (NullPointerException e) {
            // 하위 구체적인 예외 2
            System.out.println(&quot;참조 없는 객체 접근: &quot; + e.getMessage());
        } catch (Exception e) {
            // 관례상 최상위 Exception은 맨 마지막에 배치하여 인지하지 못한 나머지 예외를 모두 방어함
            System.out.println(&quot;알 수 없는 포괄적 오류: &quot; + e.getMessage());
        } finally {
            // 에러가 나든 안 나든 무조건 실행되는 영역 (자원 해제 목적)
            System.out.println(&quot;시스템 자원을 안전하게 반납합니다.&quot;);
        }
    }
}</code></pre>
<h3 id="3-예외의-위임과-의도적-발생-throws와-throw">3. 예외의 위임과 의도적 발생: throws와 throw</h3>
<p>예외를 그 자리에서 직접 처리(<code>try-catch</code>)할 것인지, 아니면 자신을 호출한 상위 메서드로 책임을 떠넘길 것인지 결정하는 키워드다. 스펠링은 비슷하지만 목적은 완전히 다르다.</p>
<table>
<thead>
<tr>
<th align="left">구분</th>
<th align="left">키워드</th>
<th align="left">목적</th>
<th align="left">적용 위치</th>
</tr>
</thead>
<tbody><tr>
<td align="left"><strong>위임 (Delegation)</strong></td>
<td align="left"><code>throws</code></td>
<td align="left">현재 메서드에서 발생한 예외의 처리를 나를 호출한 상위 메서드(Call Stack)로 떠넘김</td>
<td align="left">메서드 선언부 (파라미터 뒤)</td>
</tr>
<tr>
<td align="left"><strong>발생 (Trigger)</strong></td>
<td align="left"><code>throw</code></td>
<td align="left">개발자가 의도적으로 새로운 예외 객체를 생성하여 강제로 예외 상황을 터뜨림</td>
<td align="left">메서드 내부 로직</td>
</tr>
</tbody></table>
<pre><code class="language-java">public class ExceptionFlow {
    // 1. 위임(throws): connect()를 호출한 상위 메서드에게 예외 처리를 미룸
    public void connect() throws NullPointerException {
        checkConnection(); 
    }

    public void checkConnection() throws NullPointerException {
        // 2. 발생(throw): 특정 조건이 맞지 않을 때 개발자가 의도적으로 예외 객체를 생성해 던짐
        throw new NullPointerException(&quot;네트워크 연결 설정이 누락되었습니다.&quot;);
    }

    public static void main(String[] args) {
        ExceptionFlow client = new ExceptionFlow();
        // 최상단인 main 메서드에서 최종적으로 try-catch를 이용해 위임받은 예외를 처리함
        try {
            client.connect(); 
        } catch (NullPointerException e) {
            System.out.println(&quot;연결 실패: &quot; + e.getMessage());
        }
    }
}</code></pre>
<h3 id="4-업무-규칙-기반의-예외-설계-사용자-정의-예외-custom-exception">4. 업무 규칙 기반의 예외 설계: 사용자 정의 예외 (Custom Exception)</h3>
<p>자바가 기본으로 제공하는 문법적, 시스템적 예외(ex. 0으로 나누기, 널 포인터) 외에, 우리 프로그램만의 독자적인 '업무 규칙 위반' 상황을 예외로 정의하는 기법이다.</p>
<ul>
<li><strong>설계 목적:</strong> 은행 출금 로직에서 '잔액 부족'은 컴퓨터 입장에서는 문법 오류가 아니지만, 비즈니스 관점에서는 치명적인 예외 상황이다. 이를 명확히 표현하기 위해 사용한다.</li>
<li><strong>구현 규칙:</strong> <code>Exception</code> 클래스를 상속받아 만들며, 부모의 생성자 <code>super(message)</code>를 호출하여 오류 메시지를 세팅하는 것이 핵심이다.</li>
</ul>
<pre><code class="language-java">// 1. 시스템 예외가 아닌, 비즈니스 로직 전용 '사용자 정의 예외' 클래스 생성
public class InsufficientBalanceException extends Exception {
    public InsufficientBalanceException(String message) {
        // 부모인 Exception 클래스의 생성자를 호출하여 에러 메시지 초기화
        super(message);
    }
}

// 2. 계좌 클래스에서 업무 규칙 위반 시 명시적으로 예외 발생
class Account {
    int balance = 100;

    public void withdraw(int amt) throws InsufficientBalanceException {
        if (balance &gt;= amt) {
            balance -= amt;
        } else {
            // 잔액 부족은 시스템은 모르지만 개발자는 아는 예외이므로 강제로 throw 한다.
            throw new InsufficientBalanceException(&quot;잔액 부족으로 출금 불가&quot;);
        }
    }
}</code></pre>

<h3 id="🔥-오늘의-딥다이브-회고">🔥 오늘의 딥다이브 (회고)</h3>
<h4 id="1-사용자-정의-예외-발생-시-throws-처리가-사실상-강제되는-이유">1. 사용자 정의 예외 발생 시 <code>throws</code> 처리가 사실상 강제되는 이유</h4>
<p>오늘 실습에서 <code>InsufficientBalanceException</code>을 직접 만들고 던졌을 때, 메서드 선언부에 <code>throws</code>를 명시하거나 내부에서 <code>try-catch</code>로 묶지 않으면 컴파일 에러가 발생했다. <code>NullPointerException</code>은 굳이 처리하지 않아도 통과되는데, 왜 시스템은 내가 만든 예외 처리를 깐간하게 하는 것인가.</p>
<ul>
<li><strong>발생 원인:</strong> 내가 만든 커스텀 예외가 <code>RuntimeException</code>이 아닌 일반 <code>Exception</code>을 상속받았기 때문이다.</li>
<li><strong>시스템의 관점 (비즈니스 결함 보호):</strong> 자바는 <code>Exception</code>을 상속받은 객체를 <strong>&quot;반드시 복구 로직이 마련되어야만 하는 중대한 비즈니스적 예외&quot;</strong>로 간주한다. '잔액 부족' 같은 상황은 무시하고 지나갈 경우 데이터 무결성이 깨지는 치명적 결함으로 이어지기 때문이다.</li>
<li><strong>강력한 안전장치:</strong> 따라서 컴파일러는 개발자에게 *&quot;이 메서드는 특수 상황을 유발할 수 있으니, 호출하는 쪽에서 반드시 대비책(catch)을 마련해 둬라!&quot;* 라고 강제 서약을 받는 것과 같다.</li>
</ul>
<h4 id="2-애플리케이션-방어선-설계-checked-vs-unchecked-exception">2. 애플리케이션 방어선 설계: Checked vs Unchecked Exception</h4>
<p>자바의 모든 예외는 크게 두 가지 부류로 나뉘며, 이를 나누는 핵심 기준은 <strong>'컴파일러가 예외 처리 여부를 강제로 확인(Check)하는가'</strong>이다.</p>
<table>
<thead>
<tr>
<th align="left">구분</th>
<th align="left">컴파일 예외 (Checked Exception)</th>
<th align="left">런타임 예외 (Unchecked Exception)</th>
</tr>
</thead>
<tbody><tr>
<td align="left"><strong>상속 구조</strong></td>
<td align="left"><code>RuntimeException</code>을 제외한 모든 <code>Exception</code> 하위 클래스</td>
<td align="left"><code>RuntimeException</code> 및 그 하위 클래스</td>
</tr>
<tr>
<td align="left"><strong>확인 시점</strong></td>
<td align="left">컴파일 시점 (코드 작성 중 빨간 줄 발생)</td>
<td align="left">런타임 시점 (실행 중에 프로그램이 뻗음)</td>
</tr>
<tr>
<td align="left"><strong>처리 강제성</strong></td>
<td align="left"><code>try-catch</code>나 <code>throws</code> 처리가 <strong>필수 (강제됨)</strong></td>
<td align="left">개발자의 선택 사항 (강제 아님)</td>
</tr>
<tr>
<td align="left"><strong>대표 사례</strong></td>
<td align="left"><code>IOException</code>, 사용자 정의 예외(<code>Exception</code> 상속)</td>
<td align="left"><code>NullPointerException</code>, <code>ArithmeticException</code></td>
</tr>
</tbody></table>
<ul>
<li><strong>Unchecked Exception (강제 안 함):</strong> 주로 개발자의 실수(ex. <code>null</code> 참조, 배열 인덱스 초과)로 발생한다. 시스템 입장에서 모든 코드 라인마다 개발자의 실수를 대비해 <code>try-catch</code>를 씌우게 만들면 가독성만 해치므로 통제를 푼 것이다.</li>
<li><strong>Checked Exception (강제 함):</strong> 네트워크 단절, 파일 소실, 비즈니스 규칙 위반 등 코드를 완벽하게 짜도 외부 요인에 의해 언제든 터질 수 있는 지뢰다. 시스템의 안전을 위해 이 방어선만큼은 사전에 완벽히 대비하도록 문법적으로 강력하게 통제한다.</li>
</ul>
