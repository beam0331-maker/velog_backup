<h3 id="1-객체지향-프로그래밍oop과-자바에서의-표현">1. 객체지향 프로그래밍(OOP)과 자바에서의 표현</h3>
<p>객체지향 프로그래밍은 현실 세계의 사물(객체)을 가상 세계인 코드로 표현하는 방법론을 의미한다. 객체는 크게 '속성'과 '동작'으로 구성되며, 자바에서는 이를 <strong>클래스(Class)</strong>라는 설계도로 정의한다.</p>
<ul>
<li><strong>속성:</strong> 이름, 나이 등 데이터 ➔ <strong>인스턴스 변수</strong>로 구현</li>
<li><strong>동작:</strong> 먹기, 공부하기 등 행위 ➔ <strong>메서드(Method)</strong>로 구현</li>
<li><strong>추상화(Abstraction)</strong>: 클래스를 정의할때 불필요한 부분들을 생략하고 객체의 속성과 동장에서 필요한 부분만 추출하는 것을 의미한다.<h3 id="2-객체-생성-설계도를-실체로-만드는-new">2. 객체 생성: 설계도를 실체로 만드는 <code>new</code></h3>
</li>
<li><strong>클래스의 한계:</strong> 클래스는 단순한 설계도일 뿐이며, 스스로 동작하지 않는다.</li>
<li><strong><code>new</code> 키워드:</strong> 설계도를 바탕으로 Heap영역에 인스턴스를 생성하려면 반드시 <strong><code>new</code> 키워드</strong>를 사용해야 한다.</li>
<li><strong>객체의 실체화:</strong> <code>Cat c = new Cat();</code>과 같이 선언하여 힙 메모리를 할당받아야 비로소 프로그램 내에 객체가 실체로서 존재하게 된다.</li>
</ul>
<h3 id="3-보안을-위한-접근-지정자">3. 보안을 위한 접근 지정자</h3>
<p>객체지향의 핵심 원칙 중 하나인 정보 은닉을 위해 접근 지정자를 사용한다. 외부에서 데이터를 직접 수정하는 것을 막는것이 목적이다.</p>
<ul>
<li><strong>public:</strong> 클래스 외부 어디서든 접근이 가능하다. 주로 외부에서 사용할 수 있도록 만들어진 메서드에 부여한다.</li>
<li><strong>private:</strong> 해당 클래스 내부에서만 접근할 수 있다. 객체의 내부 데이터인 인스턴스 변수는 <code>private</code>으로 지정하여 외부로부터 직접적인 접근을 차단하는 것이 권장된다.</li>
</ul>
<h3 id="4-생성자와-메서드의-역할-구분">4. 생성자와 메서드의 역할 구분</h3>
<p><code>private</code>으로 보호된 변수를 다루기 위해 생성자와 메서드를 활용한다. </p>
<ul>
<li><strong>메서드(Method):</strong> 객체의 동작을 정의하며, 외부에서 호출해야만 실행된다.</li>
<li><strong>생성자(Constructor)와 오버로딩:</strong> 객체가 생성될 때 단 한 번 호출되어 변수를 초기화한다. 파라미터의 타입이나 개수를 다르게 하여 여러 개를 만드는 <strong>오버로딩(Overloading)</strong>이 가능하다.</li>
</ul>
<p><strong>학생 관리 어플리케이션 예시</strong></p>
<pre><code class="language-java">public class Student {
    private String name;
    private int age;
    private String address;

    // 1. 모든 정보를 입력받는 생성자 (홍길동, 이순신 등)
    public Student(String name, int age, String address) {
        this.name = name;
        this.age = age;
        this.address = address;
    }

    // 2. 주소 정보가 없는 학생용 생성자 오버로딩 (유관순 등)
    public Student(String name, int age) {
        this.name = name;
        this.age = age;
    }
}</code></pre>

<h3 id="🔥-오늘의-딥다이브-회고">🔥 오늘의 딥다이브 (회고)</h3>
<h4 id="1-변수-섀도잉-variable-shadowing과-this">1. 변수 섀도잉 (Variable Shadowing)과 <code>this</code></h4>
<ul>
<li><strong>변수 섀도잉(Shadowing):</strong> 생성자나 메서드의 파라미터(로컬 변수) 이름과 인스턴스 변수의 이름이 같을 때, 자바는 더 좁은 범위인 로컬 변수를 우선적으로 인식하는 현상이다. 이로 인해 클래스 영역에 선언된 인스턴스 변수가 가려지게(Shadowing) 된다.</li>
<li><strong><code>this</code>를 사용하는 이유:</strong> 변수 섀도잉이 발생한 상황에서, 스택(Stack) 영역의 로컬 변수가 아닌 힙(Heap) 영역에 생성된 '객체 자신'의 인스턴스 변수를 명확하게 지칭하기 위함이다. <code>this</code>는 객체 자신의 참조값을 담고 있으므로, 가려진 인스턴스 변수에 정확히 접근하여 값을 초기화할 수 있게 해준다.</li>
</ul>
<h4 id="2-자바의-파라미터-전달-방식-call-by-value">2. 자바의 파라미터 전달 방식: Call by Value</h4>
<p>자바에는 엄밀한 의미의 Call by Reference가 없다. 모든 것은 <strong>Call by Value(값에 의한 호출)</strong>로 동작한다.</p>
<ul>
<li><strong>기본형:</strong> 스택 영역에 실제 값이 복사되어 전달된다.</li>
<li><strong>참조형(객체):</strong> 객체 자체가 아니라 힙 영역의 주소값이 복사되어 전달된다. 
결국 변수가 가진 '값'을 복사해 넘긴다는 본질은 같으며, 이 구조를 이해해야 메모리 관리와 데이터 변경 흐름을 정확히 파악할 수 있다.</li>
</ul>
<h4 id="3-오버로딩과-객체지향의-이득">3. 오버로딩과 객체지향의 이득</h4>
<p>오버로딩을 통해 동일한 이름의 생성자나 메서드를 상황에 맞게 재사용함으로써 코드의 가독성이 높아진다. 이는 객체지향에서 원친중 하나인 <strong>다형성(Polymorphism)</strong>에 해당하게 된다.</p>
