<h3 id="1-람다의-축약-메서드-참조-method-reference">1. 람다의 축약: 메서드 참조 (Method Reference)</h3>
<p>람다식이 단일 메서드 호출로만 구성될 때, 파라미터를 생략하고 <code>클래스명::메서드명</code>으로 가독성을 향상시키는 기법이다.</p>
<ul>
<li><strong>유형별 분류:</strong> 정적 메서드는 <code>Integer::parseInt</code>, 인스턴스 메서드는 <code>String::toUpperCase</code> 또는 <code>System.out::println</code> 형태로 축약한다.</li>
<li><strong>생성자 참조:</strong> <code>Member::new</code>와 같이 객체 생성도 참조 형태로 줄일 수 있어 객체 생성을 담당하는 함수형 인터페이스에서 자주 쓰인다.</li>
</ul>
<h3 id="2-제네릭-와일드카드">2. 제네릭 와일드카드</h3>
<p>자바 제네릭은 와일드카드를 통해 상속 관계에 있는 타입들을 유연하게 처리할 수 있다.</p>
<ul>
<li><strong><code>? extends T</code> (상한 제한):</strong> T와 그 자식 클래스만 허용하며, 주로 데이터를 읽어오는 공급자(Producer) 역할의 리스트에 사용한다.</li>
<li><strong><code>? super T</code> (하한 제한):</strong> T와 그 부모 클래스만 허용하며, 데이터를 추가하거나 수정하는 소비자(Consumer) 역할의 리스트에 적합하다.</li>
</ul>
<pre><code class="language-java">// Pet(부모) - Cat(자식) 상속 구조 예시
List&lt;? extends Pet&gt; producers = new ArrayList&lt;Cat&gt;(); 
Pet pet = producers.get(0); // 읽기 가능 (모두 Pet의 자식이므로 안전)
// producers.add(new Cat()); // 쓰기 불가 (컴파일러가 구체적인 하위 타입을 확신할 수 없음)

List&lt;? super Cat&gt; consumers = new ArrayList&lt;Pet&gt;();
consumers.add(new Cat()); // 쓰기 가능 (최소한 Cat이거나 그 부모이므로 안전)
// Cat cat = consumers.get(0); // 읽기 불가 (어떤 부모 타입이 나올지 알 수 없어 Object로만 읽힘)</code></pre>

<h3 id="🔥-오늘의-딥다이브-회고">🔥 오늘의 딥다이브 (회고)</h3>
<h4 id="1-streaminteger-대신-intstream을-사용해야-하는-이유">1. <code>Stream&lt;Integer&gt;</code> 대신 <code>IntStream</code>을 사용해야 하는 이유</h4>
<p>객체 스트림(<code>Stream&lt;Integer&gt;</code>)은 연산 시 매번 오토박싱과 언박싱을 발생시켜 대량의 데이터 처리 시 성능 저하를 유발한다.
반면 <code>IntStream</code>은 기본형을 직접 다루어 메모리 낭비를 막고, <code>sum()</code> 등 수치 연산에 특화된 메서드를 제공하므로 훨씬 효율적이다.</p>
<h4 id="2-집계-연산의-두-갈래-intstream-집계-vs-collectors-집계">2. 집계 연산의 두 갈래: <code>IntStream</code> 집계 vs <code>Collectors</code> 집계</h4>
<p>스트림에서 합계나 평균을 구할 때 두 가지 경로가 있으며, 상황에 따른 용도 차이를 명확히 구분하는 것이 중요하다.</p>
<table>
<thead>
<tr>
<th align="left">구분</th>
<th align="left">IntStream 직접 집계</th>
<th align="left">Collectors 이용 집계 (<code>collect</code>)</th>
</tr>
</thead>
<tbody><tr>
<td align="left"><strong>대상</strong></td>
<td align="left"><code>IntStream</code>, <code>DoubleStream</code> 등 기본형 스트림</td>
<td align="left"><code>Stream&lt;T&gt;</code> 일반 객체 스트림</td>
</tr>
<tr>
<td align="left"><strong>주요 메서드</strong></td>

</tr>

<ul>
<li><strong>결론:</strong> 단순한 단일 집계는 <code>IntStream</code>으로 변환 후 처리하는 것이 빠르고, 복잡한 그룹화 작업이 병행될 때는 <code>Collectors</code>를 쓰는 것이 정석이다.</li>
</ul>
<h4 id="3-메서드-참조method-reference의-가독성-한계">3. 메서드 참조(Method Reference)의 가독성 한계</h4>
<p><code>System.out::println</code>처럼 단순 호출에는 깔끔하지만, 파라미터 순서가 바뀌거나 추가 로직이 들어오는 경우, 객체나 클래스의 이름이 길어 직관성이 떨어진다면 가독성이 떨어진다.
따라서 메서드 참조는 직관성이 높아 이해가 쉬운 경우에 적합하고, 복잡한 경우는 명시적인 람다식을 쓰는 것이 좋다.</p>
