<h3 id="2-데이터-조회-selection--projection">2. 데이터 조회: Selection &amp; Projection</h3>
<ul>
<li><strong>[개념 및 특징]</strong>
RDB의 데이터 조회는 관계 대수(Relational Algebra) 이론을 바탕으로 한다. 레코드(행)를 필터링하는 것을 Selection(σ)과 특정 속성(열)만 추출하는 것을 Projection(π)이라 부른다 .<blockquote>
<p>🌟 ** 쿼리 실행의 논리적 순서**
DB는 쿼리를 작성된 순서가 아니라 <code>FROM (테이블 선택) -&gt; WHERE (Selection 필터링) -&gt; SELECT (Projection 추출)</code> 순서로 처리하여 성능을 최적화한다.</p>
</blockquote>
</li>
<li><strong>[주요 명령어 정리]</strong><ul>
<li><code>SELECT 컬럼명</code>: Projection을 수행한다. 별칭(<code>AS</code>)을 부여하거나 문자열 리터럴을 추가해 가독성을 높일 수 있다.</li>
<li><code>FROM 테이블명</code>: 데이터를 가져올 대상 테이블을 지정한다.</li>
<li><code>WHERE 조건식</code>: Selection을 수행하여 조건이 참(<code>TRUE</code>)인 행만 통과시킨다.</li>
</ul>
</li>
</ul>
<h3 id="3-null-데이터의-특성과-방어적-처리">3. NULL 데이터의 특성과 방어적 처리</h3>
<ul>
<li><strong>[개념 및 특징]</strong>
RDBMS에서 NULL은 부재(Unknown)를 의미하며, '0'이나 '빈 칸'이 아닌 <strong>'알 수 없는 값(Unknown)'</strong>을 의미한다. NULL이 산술 연산이나 비교 연산에 포함되면 그 결과도 무조건 NULL이 되기 때문에 조회 시점이나 연산 전에 대체값으로 치환해야한다.</li>
<li><strong>[주요 명령어 정리]</strong><ul>
<li><code>IS NULL</code> / <code>IS NOT NULL</code>: 값이 NULL인지 판단한다. (주의: <code>컬럼 = NULL</code> 연산자는 동작하지 않음)</li>
<li><code>IFNULL(컬럼명, 대체값)</code>: 해당 컬럼의 값이 NULL이면 지정한 대체값으로 치환하여 연산을 안전하게 수행한다.</li>
</ul>
</li>
</ul>
<pre><code class="language-sql">-- [실무 예시] NULL 전파를 방어하는 급여 및 수당 합계 계산
SELECT 
    user_id,
    user_name,
    salary,
    comm,
    -- comm(수당)이 NULL일 경우 0으로 치환하여 연산 결과가 NULL이 되는 것을 방어함
    (salary * 12) + IFNULL(comm, 0) AS annual_income 
FROM users
WHERE comm IS NOT NULL; -- 수당이 존재하는 직원만 필터링 (Selection)</code></pre>
<h3 id="4-데이터-필터링-연산자와-다중-정렬-order-by">4. 데이터 필터링 연산자와 다중 정렬 (ORDER BY)</h3>
<ul>
<li><strong>[개념 및 특징]</strong>
논리/비교 연산자와 패턴 매칭을 통해 조건에 부합하는 튜플 집합을 구성한다. <code>ORDER BY</code>를 통한 다중 정렬은 첫 번째 정렬 기준에서 같은 값인 데이터 그룹 내에서만 두 번째 정렬 기준이 적용된다</li>
<li><strong>[주요 명령어 정리]</strong><ul>
<li><strong>범위 및 목록 조회:</strong><ul>
<li><code>BETWEEN A AND B</code>: 지정된 범위 내의 값을 찾는다. (A와 B 포함)</li>
<li><code>IN (A, B, C)</code>: 괄호 안의 목록 중 하나라도 일치하면 추출한다. (가독성 향상)</li>
</ul>
</li>
<li><strong>패턴 매칭 (<code>LIKE</code>):</strong><ul>
<li><code>%</code>: 0개 이상의 임의의 문자열 (<code>%A%</code> : A가 포함된 모든 문자)</li>
<li><code>_</code>: 정확히 1개의 임의의 문자 (<code>_A</code> : 두 번째 글자가 A인 두 글자 문자)</li>
</ul>
</li>
<li><strong>결과 정렬 (<code>ORDER BY</code>):</strong><ul>
<li><code>ORDER BY 컬럼명 ASC</code>: 오름차순 정렬 (기본값, 생략 가능)</li>
<li><code>ORDER BY 컬럼명 DESC</code>: 내림차순 정렬</li>
<li><code>ORDER BY col1 DESC, col2 ASC</code>: 다중 정렬 (col1 기준으로 내림차순 후, 값이 같으면 col2 기준으로 오름차순)</li>
</ul>
</li>
</ul>
</li>
</ul>

<h3 id="🔥-오늘의-딥다이브-회고">🔥 오늘의 딥다이브 (회고)</h3>
<h4 id="mysql-접속-시-localhost와-127001의-통신-메커니즘-차이">MySQL 접속 시 localhost와 127.0.0.1의 통신 메커니즘 차이</h4>
<p>오늘 실습 환경 설정 중 DB 클라이언트 툴에서 로컬 PC를 가리킴에도 <code>localhost</code>와 <code>127.0.0.1</code> 설정 차이로 인해 접속 에러를 겪었다. 이는 OS 계층에서 통신 프로토콜을 처리하는 방식의 차이에서 발생한 문제였다.</p>
<ul>
<li><strong>localhost (Unix Domain Socket):</strong>
MySQL 클라이언트에서 호스트를 <code>localhost</code>로 지정하면, TCP/IP 대신 <strong>Unix Domain Socket</strong>이라는 소켓(Socket) 파일 기반의 방식을 사용한다. 네트워크 계층을 거치지 않고 파일 시스템을 통해 프로세스 간 직접 데이터를 주고받는 방식이지만, 소켓 파일의 위치나 권한에 문제가 생기면 접속이 차단된다.</li>
<li><strong>127.0.0.1 (TCP/IP Loopback):</strong>
반면 IP 주소를 명시하면 실제 TCP/IP 네트워크 계층을 거쳐 통신한다. 이 경우 데이터를 쪼개고 패킷을 만들고 체크섬을 확인하며 통신이 길어지며,방화벽(Firewall) 설정이나 3306 포트 바인딩 영향을 받는다.</li>
</ul>
