<h3 id="4월-21일-트러블슈팅-잘못된-학번-입력과-트랜잭션">[4월 21일] 트러블슈팅: 잘못된 학번 입력과 트랜잭션</h3>
<h4 id="1-문제-발생-에러-메시지-누락">1. 문제 발생: 에러 메시지 누락</h4>
<p>Workshop 실습 중 실수로 존재하지 않는 학번을 입력했을때, 경고 메시지나 예외를 발생시키지 않고 루프를 계속 도는 것을 확인했다. MySQL로 확인 하지 않았으면 스스로도 작업이 완료 됐다고 생각했을 것이다.</p>
<h4 id="2-원인-분석-executeupdate의-반환값">2. 원인 분석: executeUpdate()의 반환값</h4>
<p>현재 코드가 <code>SQLException</code>으로만 예외처리하고 있던 것이 원인이다. JDBC의 <code>PreparedStatement.executeUpdate()</code> 메서드는 SQL 쿼리 문법이나 네트워크에 오류발생 시에만 <code>SQLException</code>을 던진다. 문제가 되는 오류는 조건(WHERE)에 맞는 데이터가 없어 실제 업데이트된 행(Row)이 없을 경우 <code>0</code>을 반환하는 동작으로 이 자체는 오류가 없는 동작이므로 <code>SQLException</code>이 발생하지 않는다.</p>
<h4 id="3-사용자-정의-예외처리-작성">3. 사용자 정의 예외처리 작성</h4>
<p>0 반환을 에러 상황으로 저의하기 위해 사용자 정의 예외처리를 작성한다. 대상이 없을 경우 강제로 에러를 발생시키도록 DAO 로직을 수정했다.</p>
<pre><code class="language-java">// 1. 비즈니스 논리 오류를 담당할 사용자 정의 예외 클래스
public class RecordNotFoundException extends Exception {
    public RecordNotFoundException(String message) { super(message); }
}

// 2. DAO 레이어: 데이터가 없으면 예외를 강제로 던짐 (직접 출력 금지)
public int updateStudent(Connection con, StudentDTO dto) throws RecordNotFoundException, SQLException {
    String sql = &quot;UPDATE tb_student SET ABSENCE_YN = 'Y' WHERE STUDENT_NO = ?&quot;;
    PreparedStatement pstmt = con.prepareStatement(sql);
    pstmt.setString(1, dto.getStuNo());

    int result = pstmt.executeUpdate();

    // [이론 연결] 반환값이 0인 것은 문법 오류가 아닌 '수정 대상 없음'을 의미하므로 커스텀 예외를 발생시킨다.
    if(result == 0) {
        throw new RecordNotFoundException(&quot;안내: 입력하신 학번과 일치하는 학생 정보가 없습니다.&quot;);
    }
    return result;
}</code></pre>
<h4 id="4-main으로-throws">4. Main으로 throws</h4>
<p>3-Layer 원칙을 지키기 위해, <code>RecordNotFoundException</code>을 DAO나 Service 내부에서 출력하지 않고 Main(프레젠테이션 계층)에서 출력하도록 한다.</p>
<pre><code class="language-java">// Presentation(Main) 레이어: 시스템의 모든 예외를 최종적으로 받아서 사용자에게 응답함
try {
    int n = service.updateStudentStatus(dto);
    System.out.println(&quot;성공: 학생 휴학 상태가 일괄 수정되었습니다.&quot;);
} catch (RecordNotFoundException e) {
    // 비즈니스 로직 오류 발생 시 사용자에게 친화적인 메세지를 제공한다.
    System.out.println(e.getMessage()); 
} catch (Exception e) {
    System.out.println(&quot;시스템 오류: DB 연동 중 문제가 발생했습니다.&quot;); 
}</code></pre>
<h4 id="5-데이터-무결성-보장-수동-커밋과-rollback">5. 데이터 무결성 보장: 수동 커밋과 Rollback</h4>
<p>DML 상황에서 중간에 오류가 발생할 경우를 데이터 오염을 막기위해, Service 계층에서 오토커밋을 해제하고 <code>catch</code> 블록 내에 <code>con.rollback()</code>을 명시적으로 추가한다.</p>
<pre><code class="language-java">// Service 레이어: 트랜잭션의 시작과 끝(Commit/Rollback)을 통제
public int updateStudentStatus(StudentDTO dto) throws RecordNotFoundException {
    Connection con = null;
    int result = 0;
    try {
        con = DriverManager.getConnection(url, userid, passwd);
        // [이론 연결] 명시적 트랜잭션 시작 (자동 커밋 비활성화)
        con.setAutoCommit(false); 

        result = dao.updateStudent(con, dto); 

        con.commit(); // 모든 DAO 작업이 정상 종료되면 물리적 DB에 확정
    } catch (SQLException e) {
        // [이론 연결] 시스템 오류 발생 시 이전 커밋 시점으로 데이터를 원복하여 데이터 불일치를 막는다.
        try { if (con != null) con.rollback(); } catch (SQLException ex) {} 
        throw e; // 원복 후 예외는 다시 상위로 던짐
    } finally {
        try { if (con != null) con.close(); } catch (SQLException e) {} 
    }
    return result;
}</code></pre>
<h4 id="🔥-4월-21일-딥다이브-회고">🔥 4월 21일 딥다이브 (회고)</h4>
<p><strong>오토커밋 동작 방식과 예외 기반 롤백</strong>
잘못된 학번 입력 테스트를 통해 <code>executeUpdate()</code>의 반환값을 검증하는 과정이 생략되었음을 확인했다. 오토커밋 상태에서는 트랜잭션으로 데이터 복원이 불가능하므로, 오토커밋 해제를 통해 명시적인 롤백 처리의 필요성을 확인했다.</p>

<h3 id="4월-22일-mybatis-프레임워크-도입과-데이터-문서화xml">[4월 22일] MyBatis 프레임워크 도입과 데이터 문서화(XML)</h3>
<h4 id="1-mybatis-개요-및-xml을-사용하는-이유">1. MyBatis 개요 및 XML을 사용하는 이유</h4>
<ul>
<li><strong>플랫폼 독립성과 데이터 전달</strong>
XML은 텍스트 기반의 메타데이터 형식이므로 특정 프로그래밍 언어나 운영체제에 종속되지 않는다. 따라서 데이터를 규격화하여 전달하는 역할이 가능하다</li>
<li><strong>로직의 문서화</strong>
자바 내부에 SQL이 섞여 있으면 관리가 어렵다. SQL을 XML로 분리하면 개발자가 소스 코드를 열어보지 않고도 로직을 파악할 수 있다.</li>
</ul>
<h4 id="2-외부-설정-파일-관리-properties--configuration">2. 외부 설정 파일 관리 (Properties &amp; Configuration)</h4>
<ul>
<li><strong>[JDBC 1~2단계의 환경 설정 문서화]</strong>
  기존 JDBC의 1단계(<code>Class.forName</code>)와 2단계(<code>DriverManager.getConnection</code>) 정보를 XML 문서 내의 <code>&lt;property&gt;</code> 태그로 명시하여 설정을 규격화한다.</li>
</ul>
<pre><code class="language-properties"># jdbc.properties: 접속 정보 외부화
jdbc.driver=com.mysql.cj.jdbc.Driver
jdbc.url=jdbc:mysql://localhost:3306/testdb
jdbc.userid=root
jdbc.passwd=1234</code></pre>
<pre><code class="language-xml">&lt;configuration&gt;
  &lt;properties resource=&quot;com/config/jdbc.properties&quot;/&gt;

  &lt;environments default=&quot;development&quot;&gt;
    &lt;environment id=&quot;development&quot;&gt;
      &lt;transactionManager type=&quot;JDBC&quot;/&gt;
      &lt;dataSource type=&quot;POOLED&quot;&gt;
        &lt;property name=&quot;driver&quot; value=&quot;${jdbc.driver}&quot;/&gt;
        &lt;property name=&quot;url&quot; value=&quot;${jdbc.url}&quot;/&gt;
        &lt;property name=&quot;username&quot; value=&quot;${jdbc.userid}&quot;/&gt;
        &lt;property name=&quot;password&quot; value=&quot;${jdbc.passwd}&quot;/&gt;
      &lt;/dataSource&gt;
    &lt;/environment&gt;
  &lt;/environments&gt;

  &lt;mappers&gt;
    &lt;mapper resource=&quot;com/config/DeptMapper.xml&quot;/&gt;
  &lt;/mappers&gt;
&lt;/configuration&gt;</code></pre>
<h4 id="3-sqlsessionfactory-설계와-싱글톤-패턴-적용">3. SqlSessionFactory 설계와 싱글톤 패턴 적용</h4>
<ul>
<li><strong>[Connection 관리의 추상화]</strong>
  JDBC에서 수동으로 처리하던 <code>Connection</code> 생성을 <code>SqlSessionFactory</code>가 대행한다..</li>
</ul>
<pre><code class="language-java">// static 초기화 블록을 사용하여 클래스 로드 시 1회만 XML 문서를 파싱한다.
public class MySqlSessionFactory {
    private static SqlSessionFactory sqlSessionFactory;

    static {
        try {
            String resource = &quot;com/config/configuration.xml&quot;;
            InputStream inputStream = Resources.getResourceAsStream(resource);
            // XML 설정 문서를 바탕으로 세션 생성 공장 구축
            sqlSessionFactory = new SqlSessionFactoryBuilder().build(inputStream);
        } catch (Exception e) {
            e.printStackTrace();
        }
    }

    public static SqlSession getSession() {
        return sqlSessionFactory.openSession();
    }
}</code></pre>
<h4 id="4-sql-매퍼mapper-작성-및-데이터-매핑-규칙">4. SQL 매퍼(Mapper) 작성 및 데이터 매핑 규칙</h4>
<ul>
<li><strong>[JDBC 3, 4, 6단계의 추상화]</strong>
  쿼리문 작성, 파라미터 바인딩, 결과 처리를 XML 태그를 통해 표현한다.</li>
</ul>
<pre><code class="language-xml">&lt;mapper namespace=&quot;DeptMapper&quot;&gt;
    &lt;select id=&quot;findByDeptNo&quot; resultType=&quot;com.dto.DeptDTO&quot; parameterType=&quot;int&quot;&gt;
        SELECT dept_no, dname, loc
        FROM dept
        WHERE dept_no = #{dept_no}
        &lt;/select&gt;
&lt;/mapper&gt;</code></pre>
<h4 id="5-자바-계층에서의-mybatis-호출-흐름">5. 자바 계층에서의 MyBatis 호출 흐름</h4>
<ul>
<li><strong>[JDBC 5, 7단계의 추상화]</strong>
  쿼리 실행과 자원 반납을 <code>SqlSession</code> 객체 하나로 관리한다.</li>
</ul>
<pre><code class="language-java">public com.dto.DeptDTO searchDept(int deptNo) {
    // JDBC 2단계 대체: 커넥션 대신 세션 획득
    SqlSession session = MySqlSessionFactory.getSession();
    com.dto.DeptDTO dto = null;
    try {
        // JDBC 5단계 대체: &quot;네임스페이스.쿼리ID&quot; 형식으로 호출
        dto = session.selectOne(&quot;DeptMapper.findByDeptNo&quot;, deptNo);
    } finally {
        // JDBC 7단계 대체: 개별 close 과정 없이 세션 종료로 자원 반납 처리
        session.close(); 
    }
    return dto;
}</code></pre>
<h4 id="🔥-4월-22일-딥다이브-회고">🔥 4월 22일 딥다이브 (회고)</h4>
<p><strong>[딥다이브] MyBatis namespace 태그의 개념과 식별자(ID) 충돌 방지</strong>
매퍼 XML의 루트 태그인 <code>&lt;mapper namespace=&quot;...&quot;&gt;</code>의 역할과 구조를 정리한다.</p>
<ul>
<li><strong>네임스페이스의 개념:</strong> <code>namespace</code>는 해당 매퍼 파일 내에 작성된 SQL 쿼리 ID들을 묶어주는 그룹명이다. </li>
<li><strong>설계적 필요성과 충돌 방지:</strong> 여러 테이블을 다루게 되면, 매퍼 파일마다 <code>findAll</code>이나 <code>findById</code>와 같이 동일한 이름의 쿼리 ID가 중복 생성될 수 있다. 이때 <code>namespace</code>가 없다면 MyBatis는 어떤 매퍼의 쿼리를 실행해야 할지 식별할 수 없다.</li>
<li><strong>패키지와의 유사성:</strong> 자바에서 이름이 같은 클래스를 패키지로 구분하듯, MyBatis는 네임스페이스를 통해 쿼리를 구분한다. 자바 코드에서 <code>session.selectOne(&quot;DeptMapper.findByDeptNo&quot;)</code>와 같이 <code>[네임스페이스].[쿼리ID]</code> 형태로 호출하게 강제함으로써, 식별자 충돌을 차단한다</li>
</ul>
