<h3 id="1-rowbounds를-활용한-페이징-처리">1. RowBounds를 활용한 페이징 처리</h3>
<p>MyBatis의 <code>RowBounds</code>를통해 페이징을 구현할 수 있다. 다만, 전체 데이터를 조회한 후 메모리에서 필요한 부분만 추출하므로 대용량 데이터 처리 시 주의가 필요하다.</p>
<pre><code class="language-java">// 페이징 핵심 변수 계산
int curPage = 1;      // 사용자가 요청한 현재 페이지 번호
int perPage = 3;      // 한 페이지에 보여줄 레코드 갯수
// offset: 정렬된 전체 결과에서 몇 번째 레코드부터 읽기 시작할지 결정 (0부터 시작)
int offset = (curPage - 1) * perPage; 

// session.selectList(String statement, Object parameter, RowBounds rowBounds)
// 파라미터가 없다면 null을 전달하며, RowBounds가 지정한 범위만큼만 리스트로 반환한다.
List&lt;EmpDTO&gt; list = session.selectList(&quot;com.config.EmpMapper.findAll&quot;, null, new RowBounds(offset, perPage));</code></pre>

<h3 id="2-필드명-불일치-해결-전략-resultmap-vs-as-별칭">2. 필드명 불일치 해결 전략: resultMap vs AS 별칭</h3>
<p>DB 컬럼명(<code>ename</code>)과 자바 DTO 필드명(<code>username</code>)이 다를 때 매핑하는 방법은 크게 두 가지가 있다.</p>
<h4 id="2-1-전략-1-sql-별칭as">2-1. 전략 1: SQL 별칭(AS)</h4>
<p>mapper XML에 별도의 설정을 만들지 않고, SQL 쿼리 자체에서 별칭을 부여한 <code>resultType</code> 자동 매핑이 가능하다.</p>
<pre><code class="language-xml">&lt;select id=&quot;findaAll&quot; resultType=&quot;EmpDTO&quot;&gt;
    SELECT empno,
           ename AS username,  job, sal AS salary, comm AS commission
    FROM emp
&lt;/select&gt;</code></pre>
<h4 id="2-2-전략-2-resultmap">2-2. 전략 2: resultMap</h4>
<p>매핑 규칙을 별도로 정의하여 재사용성을 높인다. JOIN 처리가 필요하거나 매핑 로직이 복잡할 때 사용한다.</p>
<pre><code class="language-xml">&lt;resultMap id=&quot;empResultMap&quot; type=&quot;EmpDTO&quot;&gt;
    &lt;id property=&quot;empNo&quot; column=&quot;empno&quot;/&gt; &lt;result property=&quot;username&quot; column=&quot;ename&quot;/&gt; &lt;result property=&quot;salary&quot; column=&quot;sal&quot;/&gt;
&lt;/resultMap&gt;

&lt;select id=&quot;findByResultMap&quot; resultMap=&quot;empResultMap&quot;&gt;
    SELECT empno, ename, sal FROM emp
&lt;/select&gt;</code></pre>

<h3 id="3-다중-파라미터-전달-방식-dto-vs-hashmap">3. 다중 파라미터 전달 방식: DTO vs HashMap</h3>
<p>기본적으로 MyBatis는 단일 객체만 파라미터로 받을 수 있다. 이때 쿼리에 전달해야 할 인자가 2개 이상일 경우 솔루션은 크게 두 가지가 있다.</p>
<h4 id="3-1-dto-방식">3-1. DTO 방식</h4>
<p>관련된 데이터가 이미 DTO로 정의되어 있거나, 데이터의 형식이 고정적일 때. DTO에 정의된 변수명에 따라 쿼리에 값이 전달된다.</p>
<pre><code class="language-xml">&lt;select id=&quot;findByDeptnoAndDname&quot; resultType=&quot;DeptDTO&quot; parameterType=&quot;DeptDTO&quot;&gt;
    SELECT * FROM dept 
    WHERE deptno = #{deptno} AND dname = #{dname}
&lt;/select&gt;</code></pre>
<h4 id="3-2-hashmap-방식">3-2. HashMap 방식</h4>
<p>서로 다른 성격의 데이터를 일회성으로 묶어야 하거나, 전용 DTO를 만들기 애매한 경우에 사용된다. <code>Map</code>의 <code>KEY</code>값에 따라 퀴리에 <code>VALUE</code> 값이 전달된다.</p>
<pre><code class="language-xml">&lt;select id=&quot;findByDeptnoAndDnameMap&quot; resultType=&quot;DeptDTO&quot; parameterType=&quot;hashmap&quot;&gt;
    SELECT * FROM dept 
    WHERE deptno = #{deptno} AND dname = #{dname}
&lt;/select&gt;</code></pre>

<h3 id="4-join-관계-매핑-속성-association--collection">4. JOIN 관계 매핑 속성 (association &amp; collection)</h3>
<h4 id="4-1-11-관계-매핑-association">4-1. 1:1 관계 매핑 (<code>association</code>)</h4>
<p>ex. 사원 정보 내에 부서 정보 객체가 하나 포함되는 경우이다.</p>
<pre><code class="language-xml">&lt;resultMap id=&quot;empDeptJoinMap&quot; type=&quot;EmpDTO&quot;&gt;
    &lt;id property=&quot;empNo&quot; column=&quot;empno&quot;/&gt;
    &lt;association property=&quot;deptInfo&quot; javaType=&quot;DeptDTO&quot;&gt; 
        &lt;id property=&quot;deptNo&quot; column=&quot;deptno&quot;/&gt;
        &lt;result property=&quot;deptName&quot; column=&quot;dname&quot;/&gt;
    &lt;/association&gt;
&lt;/resultMap&gt;</code></pre>
<h4 id="4-2-1n-관계-매핑-collection">4-2. 1:N 관계 매핑 (<code>collection</code>)</h4>
<p>ex. 부서 하나에 여러 사원이 소속되어 리스트(<code>List&lt;EmpDTO&gt;</code>) 형태로 담기는 경우이다.</p>
<pre><code class="language-xml">&lt;resultMap id=&quot;deptEmpJoinMap&quot; type=&quot;DeptDTO&quot;&gt;
    &lt;id property=&quot;deptNo&quot; column=&quot;deptno&quot;/&gt;
    &lt;collection property=&quot;empList&quot; ofType=&quot;EmpDTO&quot;&gt; 
        &lt;id property=&quot;empNo&quot; column=&quot;empno&quot;/&gt;
        &lt;result property=&quot;username&quot; column=&quot;ename&quot;/&gt;
    &lt;/collection&gt;
&lt;/resultMap&gt;</code></pre>

<h3 id="5-3-layer-아키텍처-도입과-계층별-코드-예시">5. 3-Layer 아키텍처 도입과 계층별 코드 예시</h3>
<h4 id="5-1-main">5-1. Main</h4>
<p>사용자 인터페이스와 서비스 호출을 담당한다.</p>
<pre><code class="language-java">public class DeptMain {
    public static void main(String[] args) {
        DeptService service = new DeptServiceImpl();
        service.setDao(new DeptDAO());

        List&lt;DeptDTO&gt; list = service.findAll(); // 비즈니스 계층에 데이터 요청
        list.forEach(System.out::println);
    }
}</code></pre>
<h4 id="5-2-service">5-2. Service</h4>
<p>비즈니스 로직 및 <strong>트랜잭션 경계 설정</strong>을 담당한다. <code>SqlSession</code>의 생명주기를 관리한다.</p>
<pre><code class="language-java">public class DeptServiceImpl implements DeptService {
    private DeptDAO dao;

    @Override
    public List&lt;DeptDTO&gt; findAll() {
        SqlSession session = MySqlSessionFactory.getSession(); // 트랜잭션 시작
        try {
            return dao.findAll(session); // DAO에 세션 전달
        } finally {
            session.close(); // 자원 반납 및 트랜잭션 종료
        }
    }
}</code></pre>
<h4 id="5-3-dao">5-3. DAO</h4>
<p>전달받은 세션을 통해 순수하게 DB 연동만 수행한다.</p>
<pre><code class="language-java">public class DeptDAO {
    public List&lt;DeptDTO&gt; findAll(SqlSession session) {
        // [이론 연결] JDBC의 ResultSet 루프 과정이 selectList 한 줄로 추상화됨
        return session.selectList(&quot;com.config.DeptMapper.findAll&quot;);
    }
}</code></pre>
