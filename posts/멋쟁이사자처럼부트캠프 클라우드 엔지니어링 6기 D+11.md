<h3 id="1-인터페이스와-loose-coupling-결합도-낮추기">1. 인터페이스와 Loose Coupling (결합도 낮추기)</h3>
<p>객체지향 프로그래밍에서 시스템을 설계할 때, 객체들 간의 관계를 어떻게 맺느냐에 따라 유지보수의 난이도가 달라진다. 인터페이스의 가장 주된 목적은 객체 간의 의존성, 즉 결합도(Coupling)를 낮추는 데 있다.</p>
<ul>
<li><strong>Tight Coupling (강한 결합):</strong> 한 객체가 다른 '구체적인 객체'에 강하게 의존하는 상태다. 예를 들어, 장난감이 특정 브랜드의 전용 일체형 배터리로만 동작하게 만들어진 것과 같다. 배터리를 바꾸려면 장난감 본체를 뜯어고쳐야 하듯, 코드에서도 하위 부품 클래스가 변경되면 이를 사용하는 상위 서비스 클래스의 코드까지 전부 수정해야 하는 최악의 구조다.</li>
<li><strong>Loose Coupling (느슨한 결합):</strong> 객체들이 구체적인 부품이 아니라 추상적인 '규격(인터페이스)'에 의존하는 상태다. 장난감이 'AA 사이즈'라는 표준 규격(인터페이스)만 요구한다면, 브랜드가 달라도 규격만 맞으면 어떤 건전지든 맘대로 갈아 끼울 수 있다. 코드에서도 인터페이스를 중간에 두면, 부품 클래스를 다른 것으로 교체하더라도 상위 서비스 클래스의 코드는 단 한 줄도 수정할 필요가 없다.</li>
</ul>
<blockquote>
<p>🌟 <strong>실무에서의 적용 (데이터베이스 연동 로직)</strong>
핵심 비즈니스 로직(DBService)이 특정 데이터베이스(MySQL 등)의 구체적인 클래스에 직접 의존하게 두면 안 된다. 중간에 <code>DBDAO</code>라는 인터페이스를 두어 느슨한 결합을 만들면, 추후 Oracle 등 다른 DB로 교체할 때 서비스 로직을 건드리지 않고 DB만 안전하게 갈아 끼울 수 있다.</p>
</blockquote>
<pre><code class="language-java">public class DBService {
    // ❌ [Tight Coupling의 문제] 
    // 특정 DB(OracleDAO)을 직접 명시하면, 
    // 나중에 MySQL로 바꿀 때 여기저기 코드를 다 수정해야 한다.
    // OracleDAO dao; 

    // ⭕ [Loose Coupling의 해결] 
    // 구체적인 객체 대신 'DB 연동 규격(인터페이스)'만 참조한다.
    DBDAO dao; 

    // 다형성 활용: DBDAO 규격을 지킨 객체라면 MySQL이든 Oracle이든 모두 상관없다.
    public void setDao(DBDAO dao) {
        this.dao = dao;
    }

    public void connect() {
        // 핵심 로직(DBService)은 지금 사용되는 DB를 몰라도 된다.
        // 그저 규격에 적힌 connectDB() 버튼을 누르면, 현재 힙 메모리에 생성된 실제 객체가 알아서 동작한다.
        dao.connectDB();
    }
}</code></pre>
<h3 id="2-메모리-효율과-1회성-객체-익명-클래스-anonymous-class">2. 메모리 효율과 1회성 객체: 익명 클래스 (Anonymous Class)</h3>
<p>클래스의 선언과 인스턴스 생성을 동시에 수행하는 이름 없는 클래스이다.</p>
<ul>
<li><strong>등장 배경과 메모리 이점:</strong> 단 한 번만 사용하고 버릴 객체를 위해 매번 새로운 <code>.java</code> 파일과 클래스를 만드는 것은 클래스 파일 증식과 관리 비용 낭비를 초래한다.</li>
<li><strong>추상 메서드 구현 방식의 장단점 비교 (일반 클래스 vs 익명 클래스):</strong> 상황에 따라 인터페이스를 구현하는 방식을 다르게 가져가야 메모리와 코드 관리에 유리하다.</li>
</ul>

<thead>
<tr>
<th align="left">구분</th>
<th align="left">일반 클래스 (General Class)</th>
<th align="left">익명 클래스 (Anonymous Class)</th>
</tr>
</thead>
<tbody><tr>
<td align="left"><strong>선언 위치 및 형태</strong></td>
<td align="left">별도의 <code>.java</code> 파일 생성</td>
<td align="left">메서드 내부 구현부 블록 <code>{}</code> 내에서 즉시 선언</td>
</tr>
<tr>
<td align="left"><strong>재사용성</strong></td>
<td align="left">여러 곳에서 <code>new</code> 키워드로 무한 재사용 가능</td>
<td align="left">이름이 없어 재사용(다시 <code>new</code>) 불가, 선언과 동시에 단 한 번만 생성</td>
</tr>
<tr>
<td align="left"><strong>메모리 및 유지보수</strong></td>
<td align="left">지속적으로 참조될 때 사용</td>
<td align="left">객체의 참조가 유지되는 동안만 힙(Heap) 메모리에 존재하며, 참조가 끊어지면 가비지 컬렉터에 의해 수거됨</td>
</tr>
</tbody></table>
<pre><code class="language-java">public class TestMain {
    public static void main(String[] args) {
        // 인터페이스(Flyer)는 직접 new 할 수 없지만, 
        // 중괄호 {} 열어 내부에서 추상 메서드를 즉시 구현(오버라이딩)하면 이름 없는 익명 클래스로 1회성 객체 생성이 가능하다.
        Flyer f2 = new Flyer() {
            @Override
            public void fly() {
                System.out.println(&quot;익명클래스.fly()&quot;);
            }
        };
        f2.fly(); // 생성된 1회성 인스턴스의 메서드 호출
    }
}</code></pre>
<h3 id="3-메서드의-간결한-표현-람다-표현식-lambda-expression">3. 메서드의 간결한 표현: 람다 표현식 (Lambda Expression)</h3>
<p>메서드를 하나의 식으로 표현하여 익명 클래스의 코드를 획기적으로 줄여주는 함수형 프로그래밍 기법이다.</p>
<ul>
<li><strong>적용 조건 (<code>@FunctionalInterface</code>):</strong> 람다식은 런타임 시 이름 없는 익명 구현 객체로 변환된다. 이때 컴파일러가 어떤 메서드를 오버라이딩하는지 추론해야 하므로, 반드시 <strong>오직 1개의 추상 메서드만 가진 인터페이스</strong>에서만 사용 가능하다.</li>
<li><strong>파라미터와 리턴 값 유무에 따른 4가지 표현 방식:</strong> 람다식은 파라미터(매개변수)와 리턴 값의 존재 여부에 따라 문법 요소(소괄호 <code>()</code>, 중괄호 <code>{}</code>, <code>return</code> 키워드)의 생략 규칙이 달라진다.</li>
</ul>
<table>
<thead>
<tr>
<th align="left">타입</th>
<th align="left">문법 구조 (기본형 ➔ 축약형)</th>
<th align="left">특징 및 생략 규칙</th>
</tr>
</thead>
<tbody><tr>
<td align="left"><strong>파라미터 X, 리턴 X</strong></td>
<td align="left"><code>() -&gt; { 실행문; }</code> ➔ <code>() -&gt; 실행문;</code></td>
<td align="left">매개변수가 없으면 빈 소괄호 <code>()</code>를 반드시 작성해야 한다.</td>
</tr>
<tr>
<td align="left"><strong>파라미터 O, 리턴 X</strong></td>
<td align="left"><code>(x) -&gt; { 실행문; }</code> ➔ <code>x -&gt; 실행문;</code></td>
<td align="left">매개변수가 딱 1개일 경우 소괄호 <code>()</code> 생략이 가능하다.</td>
</tr>
<tr>
<td align="left"><strong>파라미터 X, 리턴 O</strong></td>
<td align="left"><code>() -&gt; { return 값; }</code> ➔ <code>() -&gt; 값;</code></td>
<td align="left">실행문이 <code>return</code> 구문 하나뿐이면 중괄호 <code>{}</code>와 <code>return</code> 키워드를 동시에 생략한다.</td>
</tr>
<tr>
<td align="left"><strong>파라미터 O, 리턴 O</strong></td>
<td align="left"><code>(x, y) -&gt; { return x+y; }</code> ➔ <code>(x, y) -&gt; x+y;</code></td>
<td align="left">매개변수가 2개 이상이면 소괄호를 생략할 수 없다.</td>
</tr>
</tbody></table>
<pre><code class="language-java">public class LambdaTest {
    public static void main(String[] args) {
        // 1. 파라미터 X, 리턴 X (빈 소괄호 필수)
        Flyer1 f1 = () -&gt; System.out.println(&quot;f1.fly() 호출&quot;);

        // 2. 파라미터 O, 리턴 X (파라미터가 1개이므로 소괄호 생략 가능)
        Flyer2 f2 = n -&gt; System.out.println(&quot;입력값: &quot; + n);

        // 3. 파라미터 X, 리턴 O (단일 return문이므로 중괄호와 return 키워드 동시 생략)
        Flyer3 f3 = () -&gt; &quot;Hello&quot;;

        // 4. 파라미터 O, 리턴 O (파라미터가 2개 이상이므로 소괄호 필수)
        Flyer4 f4 = (n1, n2) -&gt; n1 + n2;
    }
}</code></pre>
<h3 id="4-문자열-불변성immutability과-stringbuilder의-메모리-구조">4. 문자열 불변성(Immutability)과 StringBuilder의 메모리 구조</h3>
<p>자바에서 문자열을 다룰 때, 메모리 낭비를 막기 위해 <code>String</code>과 <code>StringBuilder</code>의 내부 동작 원리를 명확히 구분하여 사용해야 한다.</p>
<ul>
<li><strong>String의 불변성 한계:</strong> <code>String</code> 클래스로 생성된 문자열 객체는 내부 값이 절대 변하지 않는다(Immutable). 따라서 문자열을 연결하거나 수정할 때마다 기존 객체를 버리고 힙 메모리에 <strong>새로운 객체를 계속해서 생성</strong>하므로 심각한 성능 저하를 유발한다.</li>
<li><strong>StringBuilder의 가변 버퍼:</strong> 문자열의 추가, 수정, 삭제가 빈번할 경우 내부의 크기가 가변적인 버퍼(Buffer)를 조작하여 <strong>하나의 객체 안에서 데이터를 변경</strong>하는 <code>StringBuilder</code>를 사용해야 메모리를 최적화할 수 있다.</li>
</ul>

<thead>
<tr>
<th align="left">주요 기능</th>
<th align="left">String (불변: 새 객체 반환)</th>

</tr>
</thead>
<tbody><tr>

<td align="left"><code>append(값)</code></td>

<td align="left"><code>insert(index, 값)</code></td>

<td align="left"><strong>문자열 삭제</strong></td>
<td align="left">지원 불가 (<code>substring()</code> 등을 활용해 우회)</td>
<td align="left"><code>delete(start, end)</code>, <code>deleteCharAt()</code></td>
</tr>
<tr>
<td align="left"><strong>문자열 뒤집기</strong></td>
<td align="left">직접 반복문을 통해 역순으로 구현해야 함</td>

</tr>
</tbody></table>
<pre><code class="language-java">public class StringBuilderTest {
    public static void main(String[] args) {
        // 내부 버퍼를 가진 StringBuilder 가변 객체 생성
        StringBuilder sb = new StringBuilder(&quot;helLO&quot;);

        // append()와 insert()를 호출해도 새로운 객체가 생성되는 것이 아니라, 원본 메모리의 데이터가 직접 조작됨
        sb.append(&quot;World&quot;).insert(0, &quot;Java_&quot;); 
        System.out.println(sb.toString()); // 조작이 끝난 후 최종적으로 사용할 때만 String 타입으로 변환
    }
}</code></pre>

<h4 id="객체-생성-패턴-creational-patterns">객체 생성 패턴 (Creational Patterns)</h4>
<p>오늘 실습에서 본 <code>Calendar.getInstance()</code>를 계기로, 정보처리기사(정처기) 디자인 패턴 파트에서 자주 출제되며 실무에서도 널리 쓰이는 대표적인 <strong>객체 생성 패턴 3가지</strong>를 찾아보고 자바 코드로 원리를 정리해 보았다.</p>
<table>
<thead>
<tr>
<th align="left">패턴 이름</th>
<th align="left">핵심 목적</th>
<th align="left">주요 사용처</th>
</tr>
</thead>
<tbody><tr>
<td align="left"><strong>싱글톤 (Singleton)</strong></td>
<td align="left">오직 하나의 인스턴스만 생성하여 자원 낭비 방지</td>
<td align="left">DB 커넥션 풀, <code>java.lang.Runtime</code></td>
</tr>
<tr>
<td align="left"><strong>팩토리 메서드 (Factory Method)</strong></td>
<td align="left">객체 생성 로직을 캡슐화하여 서브 클래스에 위임</td>
<td align="left"><code>Calendar.getInstance()</code>, 다형성 팩토리</td>
</tr>
<tr>
<td align="left"><strong>빌더 (Builder)</strong></td>
<td align="left">복잡한 객체의 생성 과정을 분리하여 가독성 확보</td>
<td align="left">매개변수가 많은 DTO 객체 생성 (<code>@Builder</code>)</td>
</tr>
</tbody></table>
<p><strong>① 싱글톤 패턴 (Singleton Pattern)</strong>
외부에서 <code>new</code>를 사용하지 못하게 막고, 시스템 전체에서 단 하나의 객체만 생성하여 메모리를 공유하는 패턴이다.</p>
<pre><code class="language-java">public class DBConn {
    // 1. 단 하나의 인스턴스를 내부에 생성
    private static DBConn inst = new DBConn();

    // 2. 외부 생성 차단을 위해 생성자를 private으로 은닉
    private DBConn() {}

    // 3. 오직 이 메서드를 통해서만 동일한 객체 주소 반환
    public static DBConn getInstance() {
        return inst;
    }
}</code></pre>
<p><strong>② 팩토리 메서드 패턴 (Factory Method Pattern)</strong>
클라이언트가 직접 구체적인 클래스를 <code>new</code> 하지 않고, 공장(Factory) 메서드에게 객체 생성을 위임하는 패턴이다. (ex. <code>Calendar.getInstance()</code>)</p>
<pre><code class="language-java">public class PetFactory {
    // 입력값에 따라 알맞은 객체를 알아서 생성 후 리턴
    public static Pet create(String type) {
        if (&quot;DOG&quot;.equals(type)) return new Dog();
        else if (&quot;CAT&quot;.equals(type)) return new Cat();
        return null; 
    }
}

// 클라이언트는 구체적인 클래스 이름(Dog)을 몰라도 됨
Pet p = PetFactory.create(&quot;DOG&quot;); </code></pre>
<p><strong>③ 빌더 패턴 (Builder Pattern)</strong>
생성자 매개변수가 많을 때 발생하는 혼동을 막기 위해, 메서드 체이닝을 활용하여 직관적으로 객체를 조립하는 패턴이다.</p>
<pre><code class="language-java">public class User {
    private String name;
    private int age;

    private User(Builder b) { // Builder를 통해서만 생성 허용
        this.name = b.name;
        this.age = b.age;
    }

    // 객체 조립을 담당할 정적 내부 클래스
    public static class Builder {
        private String name;
        private int age;

        public Builder name(String name) {
            this.name = name;
            return this; // 체이닝을 위해 자기 자신 리턴
        }
        public Builder age(int age) {
            this.age = age;
            return this;
        }
        public User build() {
            return new User(this); // 조립 완료 후 객체 반환
        }
    }
}

// 빌더 패턴 사용 예시: 어떤 값이 들어가는지 명확함
User u = new User.Builder().name(&quot;홍길동&quot;).age(30).build();</code></pre>
