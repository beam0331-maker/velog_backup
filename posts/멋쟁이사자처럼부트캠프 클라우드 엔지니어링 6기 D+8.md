<h3 id="1-객체의-동작-메서드-호출과-반환의-원리">1. 객체의 동작: 메서드 호출과 반환의 원리</h3>
<p>객체의 동작을 정의하는 메서드는 클래스 내부에 정의되어 있더라도 <strong>반드시 호출해야만 동작한다.</strong> 객체가 생성되었다고 알아서 실행되는 것이 아니며, 외부 클래스에서는 <code>객체.메서드명()</code>, 내부 클래스에서는 <code>this.메서드명()</code> 형태로 명시적인 호출이 필요하다.
호출된 메서드는 맡은 작업을 마치면 반드시 자신을 불렀던 곳으로 돌아가는데, 이를 '리턴(return)한다'고 부른다.</p>
<h3 id="2-메서드-오버로딩-캡슐화와-재사용성의-극대화">2. 메서드 오버로딩: 캡슐화와 재사용성의 극대화</h3>
<p>메서드는 외부에서 전달받는 파라미터의 개수나 타입이 다르더라도 동작이 같다면  <strong>동일한 이름으로 여러 개를 중복해서 정의</strong>할 수 있다. 이를 <strong>오버로딩(Overloading)</strong>이라고 부른다.</p>
<ul>
<li><strong>재사용성과 편의성:</strong> 예를 들어 데이터를 출력할 때 데이터 타입마다 <code>printInt()</code>, <code>printString()</code>처럼 메서드 이름을 새로 지을 필요가 없다. <code>print()</code>라는 하나의 이름표를 재사용함으로써, 호출하는 입장에서는 내부 로직을 신경 쓰지 않고 직관적으로 기능만 가져다 쓸 수 있다.</li>
</ul>
<p><strong>💡 오버로딩 예시</strong></p>
<pre><code class="language-java">public class Printer {
    // 1. 정수형 데이터를 받을 때
    public void print(int num) {
        System.out.println(&quot;입력된 숫자: &quot; + num);
    }

    // 2. 문자열 데이터를 받을 때 (타입이 다름)
    public void print(String text) {
        System.out.println(&quot;입력된 문자: &quot; + text);
    }

    // 3. 문자열과 정수를 모두 받을 때 (개수가 다름)
    public void print(String text, int num) {
        System.out.println(text + &quot; 그리고 &quot; + num);
    }
}</code></pre>
<h3 id="3-클래스-설계도-그리기-uml-표기법">3. 클래스 설계도 그리기: UML 표기법</h3>
<p>객체지향 설계 과정에서 클래스 내부의 구조를 시각화하기 위해 <strong>UML(Unified Modeling Language)</strong>의 클래스 다이어그램을 주로 사용한다. 실무에서는 코드를 짜기 전, 이 다이어그램을 통해 개발자 간에 객체의 구조를 협의한다고 하다.</p>
<p>클래스 다이어그램은 크게 <strong>클래스명, 속성, 메서드</strong> 세 구역으로 나뉘며, 데이터 보호와 캡슐화를 나타내기 위해 접근 지정자를 기호로 명확히 표기한다.</p>
<ul>
<li><code>+</code> : <code>public</code> (클래스 외부에서도 누구나 접근 가능. 주로 메서드에 사용)</li>
<li><code>-</code> : <code>private</code> (해당 클래스 내부에서만 접근 가능. 보호해야 할 인스턴스 변수에 사용)</li>
<li><code>#</code> : <code>protected</code> (동일 패키지 및 자식 클래스에서 접근 가능)</li>
</ul>
<p>표기할 때는 <code>- name: String</code> (변수명: 타입) 이나 <code>+ getName(): String</code> (메서드명(파라미터): 리턴타입) 처럼 작성하여 구조를 명세한다.</p>
<h3 id="4-메모리-관점에서의-static-키워드">4. 메모리 관점에서의 <code>static</code> 키워드</h3>
<p><code>static</code>은 객체 생성(<code>new</code>) 시점과 무관하게 동작하는 정적 키워드다.
일반적인 인스턴스 변수나 메서드는 <code>new</code>를 통해 힙(Heap) 영역에 생성되지만, <code>static</code>이 붙은 클래스, 변수, 메서드는 프로그램 시작 시점에 <strong>Method Area(메서드 영역)</strong>에 단 한 번만 단독으로 올라간다.
결과적으로 객체가 생성되기도 전인 프로그램 초기 시점부터 존재하게 되므로, <code>static</code> 메서드 내부에서는 인스턴스 변수나 생성된 객체 자신을 지칭하는 <code>this</code> 키워드를 절대 사용할 수 없다.</p>
<hr />
<h3 id="🔥-오늘의-딥다이브-회고-메서드-리턴-시점과-코딩-스타일">🔥 오늘의 딥다이브 (회고): 메서드 리턴 시점과 코딩 스타일</h3>
<p>오늘 실습한 <code>Cat.java</code> 파일의 나이 검증 메서드(<code>ageValiedCheck</code>)를 살펴보며, 동일한 논리를 처리하더라도 리턴이 발생하는 타이밍과 변수 활용 방식에 따라 코딩 스타일이 달라짐을 확인했다. 각각의 케이스가 어떤 장단점을 가지는지 정리해 보았다.</p>
<p><strong>케이스 1: 지역 변수 할당 후 마지막 단일 리턴</strong></p>
<pre><code class="language-java">private boolean ageValiedCheck(int age) {
    boolean result = false;
    if (age &gt;= 20)
        result = true;
    return result;
}</code></pre>
<ul>
<li><strong>장점:</strong> 함수의 입구와 출구가 하나씩만 존재하는 '단일 진출 원칙(SESE)'을 따른다. 마지막 <code>return result;</code> 라인에 브레이크포인트(Breakpoint) 하나만 걸어두면 디버깅 시 최종 결과값을 확인하기 매우 편하다.</li>
<li><strong>단점:</strong> 코드가 길어지고, 굳이 필요 없는 <code>result</code>라는 지역 변수를 메모리에 할당해야 한다.</li>
</ul>
<p><strong>케이스 2: if-else 분기를 통한 다중 리턴</strong></p>
<pre><code class="language-java">private boolean ageValiedCheck(int age) {
    if (age &gt;= 20) {
        return true;
    } else {
        return false;
    }
}</code></pre>
<ul>
<li><strong>장점:</strong> 조건에 따라 흐름이 명확하게 나뉘어 가독성이 좋고 직관적이다. 로직이 복잡할 경우, 조건에 맞지 않으면 불필요한 아래 코드를 실행하지 않고 즉시 함수를 빠져나갈 수 있다.</li>
<li><strong>단점:</strong> 조건식이 이미 <code>boolean</code> 결과를 가지는데 그것을 다시 <code>true</code>/<code>false</code>로 매핑하는 것은 불필요한 중복이다. 실무에서 사용하는 SonarQube 같은 정적 분석 도구에서는 경고(Code Smell)를 띄우는 안티 패턴으로 취급되기도 한다.</li>
</ul>
<p><strong>케이스 3: 연산 결과를 직접 리턴 (Direct Return)</strong></p>
<pre><code class="language-java">private boolean ageValiedCheck2(int age) {
    return age &gt;= 20;
}</code></pre>
<ul>
<li><strong>장점:</strong> 현대 프로그래밍에서 가장 선호되는 방식이다. 지역 변수 할당이나 조건문(if-else) 분기 없이 연산식 자체가 <code>boolean</code>으로 평가되어 즉시 반환되므로, 코드가 가장 간결하고 유지보수하기 좋다.</li>
<li><strong>단점:</strong> 연산 조건이 매우 복잡해질 경우(예: <code>&amp;&amp;</code>와 <code>||</code>가 여러 개 섞일 때) 중간 평가 과정을 디버깅으로 추적하기 까다로워진다.</li>
</ul>
<p><strong>💡 결론:</strong>
단순한 논리 검증이라면 <strong>케이스 3</strong>처럼 연산식을 직접 리턴하여 간결성을 확보하는 것이 제일 낫다. 하지만 검증 로직이 복잡해지거나, 중간 상태를 디버깅해야 하거나, 리턴 직전에 공통 처리가 필요하다면 <strong>케이스 1</strong>의 방식을 채택하는 것이 일반적이다.</p>
