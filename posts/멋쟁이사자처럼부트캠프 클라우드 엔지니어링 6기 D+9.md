<h3 id="1-클래스의-관계-설정-has-a포함와-is-a상속">1. 클래스의 관계 설정: Has-a(포함)와 Is-a(상속)</h3>
<p>객체지향에서 클래스 간의 관계는 크게 두 가지로 나뉜다. UML로 설계할 때 개발자 간의 소통을 위해 이를 기호로 표현한다.</p>
<ul>
<li><p><strong>Has-a (포함 관계)</strong></p>
<ul>
<li>한 클래스가 다른 클래스를 인스턴스 변수로 가지는 형태다. (ex. 트럭 has a 엔진)</li>
<li>생명 주기를 같이하는 <strong>합성(Composition)</strong>과 그렇지 않은 <strong>집계(Aggregation)</strong>로 세분화할 수 있다.</li>
<li><strong>UML 기호 표기:</strong><ul>
<li><strong>합성 :</strong> <code>전체 ◆------- 부분</code> (ex. <code>트럭 ◆------- 엔진</code> : 트럭이 폐차되면 엔진도 함께 파기됨)</li>
<li><strong>집계 :</strong> <code>전체 ◇------- 부분</code> (ex. <code>학교 ◇------- 학생</code> : 학교가 없어져도 학생은 존재함)</li>
</ul>
</li>
</ul>
</li>
<li><p><strong>Is-a (상속 관계)</strong></p>
<ul>
<li><code>extends</code> 키워드를 사용하여 기존 클래스(부모)의 자원(변수, 메서드)을 물려받아 새로운 클래스(자식)를 확장하는 형태다. 자바는 다중 상속을 지원하지 않으며 오직 <strong>단일 상속</strong>만 지원한다.</li>
<li><strong>UML 기호 표기:</strong> <code>자식 클래스 -------▷ 부모 클래스</code> (비어있는 화살표가 부모를 향함)</li>
<li><strong>자바 코드 예시:</strong><pre><code class="language-java">class Employee {
  String name;
  int salary;
}
</code></pre>
</li>
</ul>
<p>// Manager는 내부에 name, salary를 적지 않아도 Employee의 속성을 그대로 물려받음
class Manager extends Employee {</p>
<pre><code>String depart; // 자식 클래스만의 고유 속성 추가</code></pre><p>}</p>
<pre><code></code></pre></li>
</ul>
<h3 id="2-상속-부모-생성자-호출super과-메모리">2. 상속: 부모 생성자 호출(<code>super</code>)과 메모리</h3>
<p>오늘 강의에서는 상속 관계에서 객체의 생성 과정과 메모리 할당 흐름을 다루었다.</p>
<ul>
<li><strong>연쇄적인 메모리 할당:</strong> 자식 객체를 생성(<code>new Manager()</code>)하면, 힙(Heap) 영역에는 자식뿐만 아니라 부모 클래스, 나아가 최상위 클래스인 <code>Object</code>의 메모리 공간까지 모두 함께 할당된다.</li>
<li><strong><code>super()</code>의 역할:</strong> 자식에서 부모로 향하는 역방향 메모리 로드가 가능한 이유는, 자식 생성자의 첫 줄에 부모의 기본 생성자를 호출하는 <code>super()</code>가 숨겨져 있기 때문이다.</li>
<li>결론적으로 <code>new Manager()</code>라는 하나의 호출만으로도 내부적으로는 최상단 부모부터 연쇄적인 실행이 일어난다. 즉, <code>Object()</code> ➔ <code>Employee()</code> ➔ <code>Manager()</code> 순서로 부모 생성자부터 먼저 완료되며 뼈대를 잡고 내려오는 구조다.</li>
</ul>
<h3 id="3-최상위-객체-object와-tostring-재정의">3. 최상위 객체 <code>Object</code>와 <code>toString()</code> 재정의</h3>
<ul>
<li><strong>Object의 자동 상속:</strong> 자바의 모든 클래스는 개발자가 명시하지 않아도 자동으로 <code>Object</code> 클래스를 상속받는다.</li>
<li><strong>참조 변수의 기본 출력:</strong> 객체의 참조 변수를 그대로 출력하면 원래는 내부적인 메모리 주소값(ex. <code>클래스명@16진수</code>)이 반환된다.</li>
<li><strong><code>toString()</code> 오버라이딩 관례:</strong> 실무에서는 객체가 가진 상태(데이터)를 직관적인 문자열로 확인하기 위해 <code>Object</code>의 <code>toString()</code> 메서드를 오버라이딩(Overriding)하여 사용하는 것이 관례이다.</li>
<li><strong>재사용성 향상:</strong> 이 관례를 따르면 객체마다 별도의 상태 출력 메서드를 새로 만들 필요가 없어 코드 재사용성이 크게 향상된다.</li>
</ul>
<pre><code class="language-java">// Employee.java (부모 클래스)
@Override
public String toString() {
    return &quot;Employee [name=&quot; + name + &quot;, salary=&quot; + salary + &quot;]&quot;;
}</code></pre>
<h3 id="4-오버라이딩overriding과-다형성을-활용한-유연한-설계">4. 오버라이딩(Overriding)과 다형성을 활용한 유연한 설계</h3>
<p>다형성을 제대로 활용하여 구조를 유연하게 설계하기 위해서는 오버라이딩에 대한 정확한 이해가 선행되어야 한다.</p>
<h4 id="오버라이딩의-핵심-개념과-규칙">오버라이딩의 핵심 개념과 규칙</h4>
<ul>
<li><strong>개념:</strong> 부모 클래스로부터 상속받은 메서드를 자식 클래스의 상황에 맞게 내부 로직만 재정의(수정)하는 기법이다.</li>
<li><strong>규칙:</strong> 메서드의 이름, 파라미터(개수, 타입, 순서), 리턴 타입이 부모 클래스의 메서드와 완벽하게 동일해야 한다. (메서드 시그니처가 같아야 함)</li>
<li><strong>접근 지정자 제약:</strong> 자식 클래스에서 재정의할 때, 부모 메서드의 접근 지정자보다 좁은 범위로 축소할 수 없다. (예를 들어 부모가 <code>protected</code>라면 자식은 <code>protected</code>나 <code>public</code>만 가능하며, <code>private</code>으로는 변경할 수 없다.)</li>
</ul>
<h4 id="다형성의-강력함과-적용">다형성의 강력함과 적용</h4>
<p>오늘 실습에서는 여러 종류의 직군(<code>Manager</code>, <code>Engineer</code> 등)을 하나의 메서드나 배열에서 공통으로 처리해야 하는 상황을 다루었다. 이때 각 직군별로 메서드를 따로 만드는 대신, <strong>다형성을 활용하여 파라미터 타입을 부모 클래스로 지정</strong>하는 방식을 사용했다.</p>
<ul>
<li>부모 타입의 변수 하나로 다양한 자식 객체들을 모두 묶어서 관리할 수 있다.</li>
<li>이러한 특징을 활용하면 서로 다른 자식 객체들을 하나의 배열(Heterogeneous Collection)에 담거나, 모든 객체를 수용할 수 있는 범용적인 메서드를 작성할 수 있다.</li>
</ul>
<pre><code class="language-java">// 부모 타입(Employee) 파라미터 하나로 여러 자식 객체를 모두 받을 수 있다.
public static void method2(Employee e) {
    // ...
}

public static void main(String[] args) {
    // 자식 클래스인 Engineer, Manager 객체를 Employee 파라미터로 전달
    method2(new Employee(&quot;홍길동&quot;, 2000));
    method2(new Engineer(&quot;이순신&quot;, 4300, &quot;자바&quot;));
    method2(new Manager(&quot;유관순&quot;, 4000, &quot;관리&quot;));

    // Object 타입은 자바의 '모든' 데이터를 수용할 수 있다.
    method3(&quot;Hello&quot;);
    method3(3.14);
}</code></pre>

<h3 id="🔥-오늘의-딥다이브-회고">🔥 오늘의 딥다이브 (회고)</h3>
<p>오늘 배운 상속과 오버라이딩을 적용하면서, 이 문법들이 컴파일러와 실무 설계 관점에서 어떤 의미를 가지는지 추가로 고민하고 찾아본 내용을 정리한다.</p>
<h4 id="1-is-a-vs-has-a-무조건적인-상속is-a이-정답일까">1. Is-a vs Has-a: 무조건적인 상속(Is-a)이 정답일까?</h4>
<p>강의 중 '트럭 has a 엔진'과 'Manager is a Employee' 예시를 보며 상속과 합성의 차이를 배웠다. 코드를 재사용하기 위해 무작위로 상속(Is-a)을 남발하기 쉽지만, 전공 서적이나 실무 지침에서는 오히려 <strong>&quot;상속보다는 합성(Has-a)을 우선하라&quot;</strong>고 언급된다.</p>
<ul>
<li>상속은 부모와 자식 간의 결합도가 지나치게 높아져 부모 클래스가 변경되면 자식 클래스 전체가 흔들리는 문제(Fragile Base Class)가 발생할 수 있다.</li>
<li>반면 합성은 인스턴스 변수로 부모 객체를 포함시키므로, 필요할 때만 부모의 기능을 호출하여 결합도를 낮추고 유연한 설계를 가능하게 한다. 코드의 재사용성만을 위해 상속을 쓰기보다는, 정말로 <strong>의미상 완벽한 Is-a 관계일 때만 상속을 도입</strong>하는 것이 올바른 객체지향 설계임을 명심해야겠다.</li>
</ul>
<h4 id="2-override-어노테이션과-컴파일러의-소통">2. <code>@Override</code> 어노테이션과 컴파일러의 소통</h4>
<p>IDE에서 오버라이딩을 수행하면 메서드 위에 <code>@Override</code>가 자동으로 붙는다. 이를 단순한 주석이나 관례적인 표식으로만 생각했으나,<strong>메타 데이터(Meta Data)</strong> 역할을 했다는걸 알았다.</p>
<ul>
<li>만약 내가 부모 메서드의 이름을 <code>getEmployee</code>에서 <code>getEmplyee</code>로 오타를 내어 작성했다고 가정해 보자. <code>@Override</code>가 없다면 컴파일러는 이를 '자식 클래스에서 새롭게 추가한 메서드'로 인식하여 오류를 잡지 못하고 넘어가 버린다. </li>
<li>하지만 <code>@Override</code> 어노테이션이 붙어있다면, 컴파일러는 부모 클래스를 뒤져서 완벽히 똑같은 이름과 파라미터를 가진 메서드가 있는지 철저하게 검사한다. 즉, 개발자의 휴먼 에러를 컴파일 타임에 원천 차단해 주는 매우 훌륭한 안전장치인 셈이다. 앞으로 클래스를 설계할 때 제공되는 어노테이션을 적극적으로 활용하여 코드의 안정성을 높여야겠다.</li>
</ul>
