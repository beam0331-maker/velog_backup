<li><strong>JDBC(Java Database Connectivity):</strong> 자바 애플리케이션과 DBMS를 연결하는 표준 인터페이스 기술</li>
<li><strong>JDBC 7단계 절차:</strong> 드라이버 로딩부터 자원 반납까지의 프로그래밍 흐름</li>

<h3 id="1-jdbc-개요와-인터페이스-기반의-다형성">1. JDBC 개요와 인터페이스 기반의 다형성</h3>
<ul>
<li><strong>[JDBC의 개념 및 특징]</strong>
자바에서 데이터베이스에 접속하고 SQL을 실행할 수 있도록 돕는 표준 API이다. DBMS의 종류에 상관없이 동일한 메서드로 접근할 수 있도록 설계되어 있다.</li>
<li><strong>[DBMS 독립성의 원리]</strong>
자바는 표준 인터페이스(<code>java.sql</code>)를 제공하고, 각 벤더(MySQL, Oracle 등)는 이를 구현한 드라이버(<code>.jar</code>)를 제공한다. 개발자는 구체적인 구현 클래스가 아닌 인터페이스를 통해 DBMS 교체에도 수정을 최소하는, 결합도를 낮추기 위함이다</li>
</ul>
<h3 id="2-jdbc-프로그래밍-7단계-절차와-구현-예시">2. JDBC 프로그래밍 7단계 절차와 구현 예시</h3>
<ul>
<li><strong>[개념 및 특징]</strong>
데이터베이스 연동은 드라이버 로딩부터 자원 해제까지 7단계를 따른다. 외부 자원과 연동하는 과정이므로 <code>SQLException</code>과 같은 비런타임 예외 처리가 필수적이다.</li>
</ul>
<pre><code class="language-java">// [코드 예시] JDBC 7단계 표준 구현 패턴
public void executeJdbcFlow() {
    Connection con = null;
    PreparedStatement pstmt = null;
    ResultSet rs = null;

    try {
        // 1. 드라이버 로딩: MySQL 드라이버 클래스를 메모리에 적재
        Class.forName(&quot;com.mysql.cj.jdbc.Driver&quot;);

        // 2. 연결 객체 생성: DB URL 및 계정 정보를 통해 물리적 세션 확립
        con = DriverManager.getConnection(&quot;jdbc:mysql://localhost:3306/testdb&quot;, &quot;root&quot;, &quot;1234&quot;);

        // 3. SQL 작성: 실행할 쿼리문 준비 (바인딩 변수 ? 활용)
        String sql = &quot;SELECT deptno, dname, loc FROM dept WHERE deptno = ?&quot;;

        // 4. 문장 객체 생성: SQL 인젝션을 방어하고 미리 컴파일하여 성능을 높인 PreparedStatement 사용
        pstmt = con.prepareStatement(sql);
        pstmt.setInt(1, 10); // ? 위치에 값 매핑

        // 5. SQL 실행: SELECT문은 executeQuery()를 호출하여 ResultSet을 반환받음
        rs = pstmt.executeQuery();

        // 6. 결과 처리: ResultSet 커서를 이동하며 데이터를 자바 객체(DTO)로 변환
        while (rs.next()) {
            int deptno = rs.getInt(&quot;deptno&quot;);
            String dname = rs.getString(&quot;dname&quot;);
            String loc = rs.getString(&quot;loc&quot;);
            // 추출한 데이터를 DTO 객체에 담아 리스트에 추가하는 등의 작업 수행
        }
    } catch (ClassNotFoundException | SQLException e) {
        e.printStackTrace(); // 예외 발생 시 스택 트레이스 출력
    } finally {
        // 7. 자원 반납: 사용 역순으로 close() 수행하여 자원 누수(Memory Leak) 방지
        try { if (rs != null) rs.close(); } catch (SQLException e) {}
        try { if (pstmt != null) pstmt.close(); } catch (SQLException e) {}
        try { if (con != null) con.close(); } catch (SQLException e) {}
    }
}</code></pre>
<h3 id="3-sql-실행-메서드와-자원-관리-원칙">3. SQL 실행 메서드와 자원 관리 원칙</h3>
<ul>
<li><strong>[실행 메서드 구분]</strong><ul>
<li><strong>executeQuery():</strong> <code>ResultSet</code>을 반환하며 데이터 조회(SELECT) 시 사용한다.</li>
<li><strong>executeUpdate():</strong> 영향받은 레코드 개수(<code>int</code>)를 반환하며 데이터 변경(INSERT/UPDATE/DELETE) 시 사용한다.</li>
</ul>
</li>
<li><strong>[안전한 Close 규칙]</strong>
상위 객체인 <code>Connection</code>이 닫히면 하위 객체 참조 시 오류가 발생할 수 있으므로, 반드시 하위 객체부터 역순(<code>rs</code> → <code>pstmt</code> → <code>con</code>)으로 해제한다.</li>
</ul>
<h3 id="4-3-layer-아키텍처와-dto-패턴의-통합-설계">4. 3-Layer 아키텍처와 DTO 패턴의 통합 설계</h3>
<ul>
<li><strong>[책임 분리(Separation of Concerns)]</strong>
하나의 클래스에 화면 출력, 비즈니스 로직, DB 연동 코드가 섞이는 것을 방지하기 위해 역할을 계층화한다.</li>
<li><strong>[DTO(Data Transfer Object)와 계층 간 전달]</strong>
<code>ResultSet</code>은 DB 세션에 종속되어 있어 계층 간 이동 시 자원 반납을제어하기 어렵다. 따라서 데이터만을 저장하는 DTO로 변환하여 계층 간에 전달한다.</li>
</ul>

<h3 id="🔥-오늘의-딥다이브-회고">🔥 오늘의 딥다이브 (회고)</h3>
<h4 id="트러블슈팅-다중-mysqld-프로세스-충돌로-인한-접속-거부-현상-분석">[트러블슈팅] 다중 mysqld 프로세스 충돌로 인한 접속 거부 현상 분석</h4>
<p>오늘 실습 과정에서 올바른 계정 정보와 비밀번호를 입력했음에도 불구하고 JDBC를 통한 데이터베이스 접근이 반복적으로 거부(Access Denied)되는 문제를 겪었다. <code>my.ini</code> 설정 파일 등을 점검했으나 특이점이 없었고, 최종적으로 시스템 내에 여러 개의 <code>mysqld</code> 프로세스가 실행 중인 것을 발견하여 해결했다.</p>
<ul>
<li><p><strong>원인 분석 (네트워크 포트 점유와 프로세스 간섭):</strong>
운영체제 수준에서 TCP 포트(기본 3306)는 한 번에 하나의 프로세스만 점유할 수 있다. 정상 종료되지 않은 이전 버전의 <code>mysqld</code> 프로세스가 3306 포트를 차지하고 있는 상태에서 새로운 설정의 MySQL을 실행할 경우, 클라이언트는 의도한 설정(새 비밀번호 등)이 반영되지 않은 기존 '좀비' 프로세스에 연결을 시도하게 된다. 이로 인해 최신 비밀번호 설정이 무시되고 인증 실패가 발생한 것이다.</p>
</li>
<li><p><strong>해결 방법:</strong></p>
<ol>
<li>시스템 프로세스 관리 도구(<code>tasklist</code> 등)를 통해 현재 실행 중인 모든 <code>mysqld</code> 프로세스를 식별한다.</li>
<li>강제 종료 명령(<code>taskkill /f /im mysqld.exe</code>)을 통해 점유된 포트를 완전히 개방한다.</li>
<li>올바른 환경 설정이 반영된 단일 인스턴스만 재실행하여 포트 점유권을 정상화한다.</li>
</ol>
</li>
<li><p>** mysqld 프로세스란 ?**
<code>mysqld</code>는 MySQL 데이터베이스 서버의 <strong>핵심 백그라운드 데몬(Daemon)</strong>으로 다음과 같은 중추적인 역할을 수행한다.</p>
<ul>
<li><strong>네트워크 리스너:</strong> 클라이언트의 접속 요청(TCP/IP 3306 포트 등)을 대기하고 수용한다.</li>
<li><strong>인증 및 보안:</strong> 사용자의 계정 정보와 비밀번호를 검증하여 데이터 접근 권한을 관리한다.</li>
<li><strong>질의 처리(SQL Parser &amp; Optimizer):</strong> 클라이언트가 보낸 SQL 문법을 분석하고, 가장 효율적인 실행 경로(Execution Plan)를 결정한다.</li>
<li><strong>메모리 및 자원 관리:</strong> 버퍼 풀(Buffer Pool) 등을 통해 데이터와 인덱스를 메모리에 캐싱하여 입출력 성능을 최적화한다.</li>
<li><strong>저장 엔진 인터페이스:</strong> 물리적 디스크에 데이터를 저장하고 읽어오는 스토리지 엔진(InnoDB 등)과의 상호작용을 총괄한다.</li>
</ul>
</li>
</ul>

<p>네트워크 기반의 서버 프로그램은 단순히 코드뿐만 아니라 OS의 포트 바인딩 및 프로세스 상태를 함께 점검해야 함을 알았다. 특히 데몬 프로세스인 <code>mysqld</code>에 알게 되어 향후 비슷한 장애가 발생했을때 유연하게 대응할 수 있을것 같다</p>
