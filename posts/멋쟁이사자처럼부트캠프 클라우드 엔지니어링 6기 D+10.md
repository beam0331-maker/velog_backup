<h3 id="1-다형성polymorphism과-동적-타이핑">1. 다형성(Polymorphism)과 동적 타이핑</h3>
<p>하나의 참조 변수(부모 타입)로 타입이 다른 여러 인스턴스(자식 타입)를 참조하고 관리하는 객체지향의 핵심 능력이다.</p>
<ul>
<li><p><strong>업캐스팅(Upcasting):</strong> 부모 타입의 참조 변수에 자식 객체를 할당하는 것으로, 자동 형변환이 일어난다. </p>
</li>
<li><p><strong>다운캐스팅(Downcasting):</strong> 부모 타입으로 참조 중인 객체를 다시 자식 타입으로 변환하는 것으로, 자식 클래스 전용 메서드에 접근할 때 사용한다. 반드시 <code>자식타입</code>을 명시하여 강제 형변환을 해야 한다.</p>
<pre><code class="language-java">Employee e = new Manager(); // 업캐스팅
// e.b(); // 컴파일 에러: 부모 타입(Employee) 변수로는 자식 고유 기능에 접근 불가

Manager m = (Manager) e; // 다운캐스팅 (명시적 형변환)
m.b(); // 자식 고유의 메서드 정상 호출 가능</code></pre>
</li>
<li><p><strong>안전한 형변환 (<code>instanceof</code>):</strong> 무턱대고 다운캐스팅을 하면 런타임 에러(<code>ClassCastException</code>)가 발생할 수 있으므로, 실제 힙(Heap) 메모리에 생성된 객체의 타입을 먼저 확인해야 한다.</p>
<pre><code class="language-java">// 부모 타입 변수(e)가 실제로 가리키는 객체가 Manager 타입인지 안전하게 검증
if (e instanceof Manager) {
    // 검증이 통과되었을 때만 다운캐스팅 수행
    Manager m = (Manager) e;
    m.b(); // 다운캐스팅 후 자식 고유의 메서드 사용 가능
}</code></pre>
</li>
</ul>
<blockquote>
<p>**    다형성을 활용한 파라미터 설계**
메서드의 파라미터를 부모 타입으로 지정하면, 각각의 자식 객체마다 오버로딩을 할 필요 없이 하나의 메서드로 모든 자식 객체를 수용할 수 있어 코드 재사용성이 극대화된다.</p>
</blockquote>
<pre><code class="language-java">public class TestEmployee {
    // 다형성이 적용된 파라미터: Employee를 상속받은 모든 객체(Manager, Engineer)를 수용한다.
    public static void method2(Employee e) {
        // 실제 메모리에 올라간 객체가 Manager 타입인지 검증 (안전성 확보)
        if (e instanceof Manager) {
            // 자식 클래스 고유의 기능에 접근하기 위한 다운캐스팅
            Manager m = (Manager) e;
            m.a(); 
        }
    }

    public static void main(String[] args) {
        method2(new Manager(&quot;유관순&quot;, 4000, &quot;관리&quot;)); 
        method2(new Engineer(&quot;이순신&quot;, 4300, &quot;자바&quot;));
    }
}</code></pre>
<h3 id="2-패키지와-접근-지정자를-통한-데이터-은닉">2. 패키지와 접근 지정자를 통한 데이터 은닉</h3>

<h3 id="3-추상-클래스abstract-class를-통한-규격-강제">3. 추상 클래스(Abstract Class)를 통한 규격 강제</h3>
<p>메서드의 선언부만 있고 구현부<code>{}</code>가 없는 추상 메서드를 하나라도 포함하는 클래스이다.</p>
<ul>
<li><strong>미완성 설계도:</strong> 구현되지 않은 메서드가 존재하므로 <code>new</code> 키워드를 통한 직접적인 인스턴스 생성이 불가능하다.</li>
<li><strong>오버라이딩 강제성:</strong> 추상 클래스를 상속받은 하위 클래스는 반드시 부모의 추상 메서드를 구체화(재정의)해야만 컴파일 에러를 피할 수 있다. 이를 통해 개발자 간 메서드 명명 규칙과 기능을 통일할 수 있다.</li>
</ul>
<pre><code class="language-java">// 추상 키워드(abstract) 명시: 이 클래스는 직접 new 할 수 없다.
public abstract class Employee {
    String name;
    int salary;

    // 추상 메서드: 자식 클래스가 반드시 본인 상황에 맞게 구현해야 하는 '규격'을 제시
    public abstract String getEmployee(); 
}

public class Manager extends Employee {
    String depart;

    // 부모의 추상 메서드를 오버라이딩하여 구현부를 완성해야 한다. (강제성)
    @Override
    public String getEmployee() {
        return name +&quot;\t&quot;+ salary+&quot;\t&quot; + depart;
    }
}</code></pre>
<h3 id="4-인터페이스interface를-통한-다중-구현과-역할-분리">4. 인터페이스(Interface)를 통한 다중 구현과 역할 분리</h3>
<ul>
<li><strong>다중 구현:</strong> 자바의 단일 상속 한계를 극복한다. 하나의 클래스는 상속(<code>extends</code>)과 동시에 여러 인터페이스를 다중 구현(<code>implements</code>)할 수 있다.</li>
<li><strong>역할 분리와 선택적 기능 부여:</strong> 만약 비행 기능을 부모인 <code>Pet</code> 클래스에 선언해 버리면, 하늘을 날지 못하는 강아지나 고양이까지 억지로 비행 기능을 상속받아 구현해야 하는 모순이 발생한다. 따라서 '날 수 있는 행위'를 별도의 인터페이스로 분리해 내고, 비행이 가능한 클래스에만 선택적으로 <code>implements</code>하여 기능을 장착하는 것이다.</li>
</ul>
<pre><code class="language-java">// [잘못된 설계] 추상 클래스에 기능을 억지로 우겨넣을 경우
public abstract class BadPet {
    public abstract void eat();
    public abstract void fly(); // 모든 애완동물은 날아야 한다는 강제 규격이 생겨버림!
}

public class Dog extends BadPet {
    @Override public void eat() { /* 강아지가 밥을 먹음 */ }

    // 강아지는 날 수 없는데 컴파일 에러를 피하기 위해 억지로 구현해야 하는 모순 발생!
    @Override public void fly() { 
        // 설계 훼손
    }
}


// ==========================================


// [올바른 설계] 인터페이스를 활용한 독립적인 역할 부여
public interface Flyer {
    public abstract void fly(); // 날 수 있는 행위만 따로 떼어내어 모듈화
}

// Bird는 Pet으로서의 정체성(상속)을 유지하면서, Flyer라는 부가적인 행위(구현)를 선택적으로 장착한다.
public class Bird extends Pet implements Flyer {
    // Pet의 추상 메서드 구현 (정체성)
    @Override
    public void eat() {}

    // Flyer의 추상 메서드 구현 (새만의 비행 방식 구체화)
    @Override
    public void fly() {
        // 날아가는 기능 구체화
    }
}</code></pre>
